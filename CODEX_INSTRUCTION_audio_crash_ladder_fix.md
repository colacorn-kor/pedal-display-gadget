# 건① 오디오 태스크 폭주 → 워치독 리셋 (최우선)

## 증상 (실기 로그)
```
W (885153) app: I2S RX queue overflow count: 509      ← 90ms마다 계속 증가
E (890073) task_wdt: Task watchdog got triggered ... IDLE1 (CPU 1)
E (890073) task_wdt: CPU 1: audio
--- 0x4200ba2f: audio_task at app_main.c:292           ← ESP_LOGW 안에서 멈춤
--- 0x4200ce06: analyze at tuner.c:184                 ← 두 번째 백트레이스
```

## 원인 분석 (측정·계산으로 확인됨)
- `analyze()`는 tau 5~268 × 내부 `WIN-tau` 루프 = **369,468회 반복** → 240MHz에서 **14~18.5ms**.
- 실행 주기는 `HOP 192 @ 8kHz` = **24ms** → **CPU 점유율 58~77%**.
- I2S DMA 버퍼는 `dma_desc_num=4 × DMA_FRAMES 256` = **21.3ms 분량**뿐.
  → `analyze()` 1회가 버퍼를 거의 다 소진 → 오버플로.
- 오버플로가 시작되면 `i2s_channel_read`가 **즉시 반환**(큐에 항상 데이터) →
  우선순위 6의 `audio_task`가 블로킹 없이 회전 → **IDLE1 굶주림 → 워치독**.
- 게다가 오버플로 로그가 **카운트 변경마다** 발행된다. 115200baud에서 한 줄이 4ms 이상
  태스크를 붙잡아 상황을 악화시킨다(첫 백트레이스가 `uart_write` 안에서 잡힌 이유).

## ①-A 오버플로 로그 레이트 제한 (즉시 효과)
`app_main.c` 292행 부근. 현재 `overflows != reported_overflows`이면 매번 로그한다.
- **최소 2000ms 간격**으로 제한한다. 마지막 로그 시각을 정적 변수로 두고 비교.
- 억제된 구간의 정보가 사라지지 않도록 **누적 카운트와 증가분(delta)을 함께** 출력:
  `"I2S RX overflow: total=%u (+%u in last %ums)"`.
- 같은 함수의 `"I2S returned an invalid byte count"` 로그도 동일하게 2초 제한을 건다
  (오류 상황에서 같은 폭주가 가능).

## ①-B DMA 버퍼 증설
- `chan_cfg.dma_desc_num = 4` → **8**.
- 버퍼가 21.3ms → 42.6ms가 되어 `analyze()` 최악 18.5ms에 24ms의 여유가 생긴다.
- 추가 메모리 8 × 256 × 4B = 8KB(내부 RAM). 허용 범위.

## ①-C `analyze()` 실행 빈도 완화
- `tuner.c`의 `#define HOP 192` → **384**.
- analyze 주기가 24ms → 48ms가 되어 CPU 점유율이 **58~77% → 29~39%** 로 떨어진다.
- 튜너 갱신율은 41.7Hz → 20.8Hz. 튜너 UI로는 충분히 빠르다.
- ⚠ `tuner.c`는 sim과 공유하는 파일이다. **상수 1개만** 바꾸고 알고리즘·다른 상수는 건드리지 말 것.
  `_Static_assert` 3개가 그대로 통과하는지 확인할 것.

## ①-D 워치독 안전망 (yield)
①-B/C로 정상 상황에서는 `i2s_channel_read`가 블로킹하므로 IDLE이 정상 동작한다.
그래도 이상 상황에서 폭주가 재발하지 않도록 안전망을 둔다:
- `audio_task` 루프에 처리 블록 카운터를 두고, **64블록마다(≈341ms) `vTaskDelay(1)`** 를 호출.
- 오디오 처리 지연은 무시할 수준이며, IDLE1이 반드시 실행되어 워치독이 트리거되지 않는다.

## 건① 완료 판정
- 튜너 앱을 **5분 이상** 켜둬도 워치독 리셋 없음.
- 오버플로 로그가 나오더라도 **2초에 1줄 이하**.
- 정상 상태(오버플로 0)에서 튜너 음정 표시가 종전과 동일하게 동작.
- 다른 앱(monitor/images/bounce) 거동 무변경.

# 건② TRS 래치 — 과도 스윕 오래치 수정

## 증상 (실기 로그)
```
I (281493) ladder ratio=0.1104 -> DEADZONE (latch=HOME)   ← LEFT를 눌렀는데 HOME이 래치됨
I (283493) ladder ratio=0.0948 -> LEFT     (latch=HOME)   ← LEFT로 디코드돼도 래치가 안 바뀜
```
UP을 빠르게 연타했을 때 의도치 않게 앱이 실행된 것도 이 버그다(과도 구간에서 OK가 래치됨).

## 원인
무입력(ratio≈1.0)에서 키를 누르면 전압이 **위에서 아래로 쓸고 내려간다.** 32회 평균이 그
과도 구간에 걸치면 중간값이 나오고, 목표 키보다 위쪽 키(HOME·OK) 창에 먼저 걸린다.
그런데 현재 정책이 **"더 높은 비율로만 전환"** 이라 아래쪽 목표 키로 내려가는 것을 막아,
잘못된 래치가 그대로 고정된다.

## ②-A 래치 갱신에 연속 안정성 요구
`input_ladder_poll()`의 래치 전이 로직에 **연속 확인**을 추가한다.
- 후보 상태(`decoded` 또는 `idle`)가 **3회 연속(=30ms, `INPUT_POLL_MS` 10ms 기준)** 동일할 때만
  래치를 갱신한다. 카운터와 직전 후보를 정적 변수로 유지.
- 과도 구간은 1폴 이내로 지나가므로 완전히 걸러진다.
- **기존 전이 규칙 4가지는 그대로 유지**한다(안정성 조건이 통과한 뒤에 적용):
  1. 래치 없음 + 유효 키 → 래치
  2. 래치 있음 + idle → 해제
  3. 래치 있음 + **더 높은 비율**의 다른 키 → 전환
  4. 래치 있음 + 더 낮은 비율 / 데드존 → 무시(현재 키 유지)
- `#define INPUT_TRS_STABLE_COUNT 3` 으로 상수화.

⚠ 이 안정성 카운터는 **래치 갱신에만** 적용한다. 기존 `input_button_update()`의
디바운스/홀드/리피트 상태머신은 **절대 수정하지 말 것**.

# 건③ 판정표 재보정 (하드웨어 변경 반영)

## 회로 변경 완료 (2026-07)
접점 저항(택트 스위치+브레드보드, 0~100Ω)이 작은 키 저항에서 큰 오차를 유발해
UP이 전혀 인식되지 않았다. Rtop을 키워 접점 저항 비중을 낮춘 새 회로로 교체 완료:

```
Ring(+3V3) ─[Rtop 10k]─┬─→ Tip ──→ G4 (ADC1_CH3)
                        │
   UP    ─ 직결(0Ω) ─ Sleeve      DOWN ─ [470Ω] ─ Sleeve
   LEFT  ─ [1k]     ─ Sleeve      RIGHT─ [2k]   ─ Sleeve
   OK    ─ [4.7k]   ─ Sleeve      HOME ─ [10k]  ─ Sleeve
```
- UP↔DOWN 간격이 **32mV → 112mV** 로 개선. 모든 인접 간격 112mV 이상.
- 최대 전압 1.61V(ADC 선형 구간), 소비 전류 0.33mA.

## ③-A 판정표 교체 (잠정값)
`app_main.c`의 6개 CENTER/WINDOW 상수를 아래로 교체한다.

| 키 | CENTER | WINDOW | 참고 @3208mV |
|---|---|---|---|
| UP | 0.0051 | 0.0117 | 0 ~ 54 mV |
| DOWN | 0.0505 | 0.0096 | 131 ~ 193 mV |
| LEFT | 0.0970 | 0.0101 | 279 ~ 344 mV |
| RIGHT | 0.1736 | 0.0387 | 433 ~ 681 mV |
| OK | 0.3285 | 0.0400 | 926 ~ 1182 mV |
| HOME | 0.5113 | 0.0400 | 1512 ~ 1769 mV |
| IDLE | ratio ≥ 0.85 | — | ≥ 2727 mV |

- 접점 저항 0~100Ω + 이전 실측에서 확인된 ADC 상향 편의(최대 +4%)를 반영한 값이다.
- 57개 조합 검증: 2키 조합 오인식 **0건**. 3키 이상 2개 조합(`LEFT+RIGHT+OK` 계열)이
  DOWN으로 읽히지만 실사용에서 발생하지 않는 입력이므로 허용한다.
- 각 상수 옆에 유래(저항값·예상 전압)를 주석으로 남길 것.

## ③-B 로깅 유지
`INPUT_TRS_LOG`는 **1로 유지**한다. 새 회로의 실측 재보정이 아직 남아 있다.
로그 포맷은 현행 유지(raw / mV / ratio / 판정 / latch / idle_ref).

## 건②③ 완료 판정
1. 6키 각각 정확 인식. **특히 UP이 안정적으로 인식**되어야 한다(이번 수정의 핵심).
2. UP을 빠르게 연타해도 다른 키 이벤트나 앱 전환이 **발생하지 않음**.
3. 임의의 키를 눌렀을 때 로그의 `latch`가 **처음부터 해당 키**로 잡힘(HOME/OK 경유 없음).
4. `RIGHT+OK` 동시 입력 → 이벤트 없음(데드존).
5. 키를 누른 채 다른 키 추가 → 처음 키 유지.
6. 방향키 오토리피트, HOME 롱프레스(런처 복귀), FOOTSW 단/장 누름 정상.
